# BUG 激活原因设计

## 目标

调整 BUG 详情“处理信息”的展示顺序，并在状态变更为“被激活”时强制填写激活原因，使当前原因可在详情查看、历次原因可在变更历史追溯。

## 页面设计

- 详情“处理信息”顺序固定为：指派给、修复时间、解决方案、关闭时间、激活原因。
- 解决方案必须位于关闭时间之前。
- 激活原因独占一行，仅当前状态为“被激活”时展示；其他状态不展示该字段，避免出现过期原因或无意义占位符。
- 状态变更目标为“被激活”时，在统一状态弹窗中增加多行文本“激活原因”，必填，去除首尾空格后长度为 1 至 500 字。
- BUG 列表不新增激活原因列。

## 数据与状态规则

- `pms_bug` 新增 `activation_reason TEXT`，可空。
- 状态变更为 `3`（被激活）时，必须保存非空激活原因，同时清空修复时间、关闭时间和解决方案。
- 从“被激活”变更为其他状态时，清空当前记录的激活原因；对应旧原因仍保留在操作日志中。
- migration 为现有“被激活”的模拟 BUG 回填“问题再次复现，重新激活处理”，再增加约束：状态为被激活时原因必填，其他状态原因必须为空。
- 每次状态变更的 `status`、`activation_reason` 及其他处理字段共享同一个 `operation_id`。

## 接口与历史

- 状态接口新增读取 `activation_reason`，缺少或超过 500 字时返回明确错误。
- BUG 详情接口返回 `activation_reason`，前端 API 转换为 `activationReason`。
- 变更历史字段顺序把 `activation_reason` 放在 `status` 之后，中文名称为“激活原因”。
- 多次激活时，详情只显示当前一次原因；历史按每次状态操作分别展示全部原因。

## 验收

- 被激活弹窗没有激活原因时不能提交，填写后可以提交。
- 被激活详情显示激活原因且独占一行；解决方案位于关闭时间之前。
- 离开被激活状态后详情不显示激活原因，但历史仍能看到旧原因。
- 初始化 SQL、migration、Markdown 和 Excel 表结构说明保持一致。
