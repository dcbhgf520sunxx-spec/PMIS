# 需求查询路径与状态联动设计

## 目标

需求列表查询中，“状态”选项跟随“需求路径”联动，避免用户组合出不属于该路径的状态条件。

## 交互规则

- 未选择需求路径时，状态下拉展示全部需求状态。
- 选择需求路径后，状态下拉只展示该路径可能出现的状态。
- 四条路径均包含公共处理状态：需求整理、实施中、试运行、已完成、已完成未使用、暂停。
- 上会立项额外包含：上会评估、需求上会、上会通过、过会未通过。
- 需求提报额外包含：提报评估、需求审批、审批通过、审批未通过。
- 预研额外包含：需求验证、预研通过、预研不通过。
- 直接实施没有额外前置状态。
- 更换需求路径时，如果当前已选状态不属于新路径，自动清空状态；属于新路径则保留。
- 重置查询仍使用现有统一重置能力，同时清空路径和状态。

## 实现范围

- 复用需求模块现有状态标签和路径定义，在前端生成路径对应的状态选项。
- 需求列表筛选区只负责根据草稿路径计算选项，并在路径变化时校正草稿状态。
- 后端查询接口不变，继续接收 `requirement_type` 和 `status` 组合条件。
- 不新增公共组件、不新增接口、不修改数据库结构。

## 验收

- 自动测试覆盖四条路径的状态集合、公共状态和路径切换清空逻辑。
- 运行前端测试、严格组件审计、接口契约审计、前端构建和统一交付门禁。
- 浏览器实际验证未选路径、选择路径、更换路径及查询结果。
