# PMIS 正式环境替换与数据迁移设计

## 目标与约束

在不重装、不更换现有服务器 `172.16.0.148` 的前提下，将当前 PMIS 部署为新的正式项目管理系统，并把旧 `project-manage-system` 中仍需使用的业务数据从 MySQL 迁移到 PostgreSQL。旧系统在新系统完成验证前保持可恢复，不直接覆盖或删除。

固定约束如下：

- 服务器继续使用 CentOS 7，不安装 Docker，不升级或重装操作系统。
- PostgreSQL 16 和 Node.js 22 使用官方源码安装到独立目录，不覆盖旧 MySQL、旧 Node.js 16 或旧 PM2 进程。
- 旧系统继续使用 `/opt/project-manage-system`、MySQL `3306`、后端 `3001` 和前端 `8080`，直到正式切换。
- 新系统使用 `/opt/pmis`、PostgreSQL `5433`、后端 `3103` 和验收入口 `3104`。
- 不迁移里程碑数据，不迁移 MCP 审计数据，不在新 PMIS 中保留任何 MCP 或 Agent 能力。
- 当前无人使用旧系统，正式迁移采用停机后的最终全量迁移，不建设双写、增量同步或在线迁移链路。

## 服务器运行架构

PostgreSQL 16 安装在 `/opt/postgresql-16`，数据目录使用 `/var/lib/pgsql/16/data`，由独立系统用户和 systemd 服务管理。数据库仅监听 `127.0.0.1:5433`，不向局域网开放数据库端口。生产数据库名为 `pmis`，应用使用独立数据库账号连接。

Node.js 22 安装在 `/opt/node-v22`，不替换系统现有 `node`、`npm` 或旧 PM2。新后端由独立的 `pmis-backend.service` 启动，显式调用 `/opt/node-v22/bin/node /opt/pmis/backend/src/server.js`，监听 `127.0.0.1:3103`。这样旧 `pms-backend` 继续使用 Node.js 16，两套运行时互不影响。

前端在本地通过当前项目工具链完成构建，只把验证过的 `frontend/dist` 发布到服务器。Nginx 增加临时验收入口 `3104`，静态目录指向 `/opt/pmis/frontend/dist`，`/api/` 和 `/uploads/` 代理到 `127.0.0.1:3103`。正式切换时将现有 `8080` 入口的静态目录和 API 代理改到新 PMIS；旧 Nginx 配置保留备份，可直接恢复。

CentOS 7 不属于当前 Node.js 22 和 PostgreSQL 16 的标准二进制支持环境，因此源码安装必须先经过可行性门禁：编译、安装、启动、依赖安装、后端测试和健康检查全部通过后，才允许进入数据迁移。任一门禁失败时停止部署，不修改旧系统入口，也不通过非官方二进制或降低安全校验强行继续。

## 发布版本与目录

正式发布必须来自一个明确的 Git 提交，不直接上传当前存在未提交改动的工作目录。发布前先完成当前 PMIS 功能收口、数据库文档同步、自动门禁、前后端测试、构建和本地浏览器验收，再生成发布包。

服务器目录分工如下：

- `/opt/pmis/releases/<commit>/`：按 Git 提交保存的只读发布版本。
- `/opt/pmis/current`：指向当前发布版本的软链接。
- `/opt/pmis/shared/backend.env`：生产环境变量，权限为 `600`，不进入 Git。
- `/opt/pmis/shared/logs/`：新后端日志。
- `/opt/backups/pmis/`：PostgreSQL 备份、迁移报告和切换前快照。

发布切换只调整 `current` 指向并重启新后端，不在发布目录中现场修改源码。每次发布保留上一版本，以便应用代码快速回退。

## 数据迁移范围

### 迁移数据

以下数据保留旧主键和原始创建、更新时间：

- `pms_user`：20 条。保留员工号、姓名、手机号、状态、首次登录标记和原 bcrypt 密码哈希，用户不需要统一重置密码。
- `pms_role`：2 条。保留 `admin` 和 `JS_001` 两个角色。
- `pms_user_role`：21 条。保留原用户角色关系。
- `pms_archive_type`：1 条。
- `pms_archive`：11 条任务类型档案。
- `pms_product`：5 条。
- `pms_project`：10 条。
- `pms_requirement`：37 条。
- `pms_task`：31 条，其中包含 2 条软删除记录；软删除状态继续保留。
- `pms_operation_log`：178 条，转换写入新表 `pms_op_log`。

旧库中的 Bug 和工单均为 0 条，不产生业务数据导入，但新系统继续保留对应空表和功能。

### 转换规则

- 旧 `pms_project.member_ids` 按逗号拆分、去空值、转成数字并去重，再写入 `pms_project_member`。当前 8 个项目包含成员，预计生成 18 条成员关系；导入前必须验证所有成员 ID 都存在于 `pms_user`。
- 旧 `pms_operation_log` 写入 `pms_op_log`，保留操作人、动作、模块、目标、字段、新旧值、IP 和时间。旧日志没有 `operation_id`，历史记录按单条操作保留，不伪造跨字段操作分组。
- MySQL `DATETIME` 按 `Asia/Shanghai` 解释后写入 PostgreSQL `TIMESTAMPTZ`，日期字段保持原日期，不进行时区偏移。
- 任务 `task_type` 保持原档案 ID；项目、需求、负责人、创建人和更新人关系保持原 ID。
- 导入完成后，将所有 `BIGSERIAL` 序列重置到对应表最大 ID，避免新建记录主键冲突。

### 权限转换

新 PMIS 使用当前初始化脚本中的菜单，不复制旧 `pms_menu` 主键或路径。角色权限按角色 code 和新菜单 code 重建：

- `admin`：授权所有启用的新菜单。
- `JS_001`（业务权限）：授权首页、产品、项目、需求、任务、Bug 和工单；不授权基础设置、用户权限和组件工作台。

用户、角色和用户角色关系保留，旧 `pms_role_menu` 不直接导入。这样可避免旧 Vue 路由和新 React 路由错配。

### 明确舍弃

- `pms_milestone` 的 7 条数据及相关操作记录。
- `pms_mcp_audit_log` 的 643 条记录。
- MCP 路由、鉴权、工具定义、客户端配置和 Agent 接入能力。
- 旧菜单和旧角色菜单关系。
- 本地开发数据库中的测试、演示业务数据。

舍弃的数据仍随最终 MySQL 全量备份归档，但不进入新 PostgreSQL，也不在新系统页面展示。

## 迁移工具

新增一次性迁移工具，独立放在 `tools/legacy-data-migration/`，只承担旧 MySQL 到新 PostgreSQL 的数据读取、转换、写入和核验，不进入生产后端运行依赖。工具使用独立的 `package.json` 管理 MySQL 和 PostgreSQL 驱动，连接信息只从受限环境文件读取，日志不得输出密码、密码哈希、手机号或完整业务描述。

迁移分为四个阶段：

1. `precheck`：只读检查源表数量、必填字段、枚举值、外键引用、项目成员格式和目标库状态。目标库存在非初始化业务数据时拒绝执行。
2. `import`：在一个 PostgreSQL 事务中按用户、角色、用户角色、档案、产品、项目、项目成员、需求、任务、操作日志的顺序导入；任一步失败则整体回滚。
3. `sequence`：事务内重置相关序列。
4. `verify`：重新读取源库和目标库，输出不含敏感值的逐表数量、关系数量、舍弃数量和一致性结果。

迁移工具必须支持对独立演练数据库重复执行，但不提供自动清空正式库的参数。正式库如需重做，必须删除并重新创建该数据库，避免隐藏式覆盖已导入数据。

## 演练与数据验收

正式迁移前使用旧库最新备份创建演练源库，并创建全新的演练 PostgreSQL 数据库。演练至少验证：

- 源库和目标库的迁移表数量逐一一致。
- 18 条项目成员关系全部生成且无重复、无孤儿用户。
- 产品、项目、需求、任务的负责人及上下游关联无孤儿记录。
- 需求状态、任务状态、任务类型、优先级和软删除值均在新系统允许范围内。
- 用户密码哈希保持原值，管理员和业务角色各完成一次真实登录。
- 管理员可访问全部菜单；业务权限用户只能访问约定业务菜单，受限页面和接口均被拒绝。
- 产品、项目、需求、任务列表和详情能读取迁移数据，历史记录能正常展示。
- 新增一条演练记录后主键大于迁移数据最大 ID。
- 新系统不存在里程碑入口、MCP 路由和 MCP 数据。

演练结果形成机器可读 JSON 和简明 Markdown 报告，保存源数量、目标数量、失败项和执行时间。任何数量或关联校验不一致时不得进入正式切换。

## 正式切换

当前无人使用旧系统，因此正式切换不建设在线增量迁移。执行顺序固定如下：

1. 保存旧应用目录、Nginx 配置、PM2 状态、crontab 和 MySQL 全量备份，并记录文件校验值。
2. 停止旧 `pms-backend`，保留 MySQL 运行，确认旧库不再产生新业务写入。
3. 执行最终 MySQL 全量备份。
4. 创建全新的正式 PostgreSQL 数据库，执行当前初始化 SQL并登记 migration 基线。
5. 运行迁移工具的 `precheck`、`import` 和 `verify`。
6. 启动新后端，检查 `3103/api/health`、登录、权限和核心业务页面。
7. 将 Nginx `8080` 入口切换到新前端和新后端，检查配置后平滑重载。
8. 从实际访问入口再次完成健康检查和核心业务冒烟验收。

正式切换前不删除旧数据。新系统验收通过后，旧后端保持停止状态，旧 MySQL 和旧应用目录至少保留 30 天。

## 回滚设计

以下任一情况触发回滚：迁移数量不一致、关键外键缺失、用户无法登录、权限越权或缺失、核心页面无法读取数据、Nginx 切换后健康检查失败。

回滚时恢复切换前 Nginx 配置并平滑重载，重新启动旧 `pms-backend`，确认旧 `3001` 健康检查和 `8080` 页面恢复。旧 MySQL 在整个迁移期间不做结构修改，因此回滚不需要反向同步 PostgreSQL 数据。新 PostgreSQL 和新应用保留用于排查，不覆盖旧库。

由于切换期间旧系统已停止写入，回滚不存在新旧系统数据合并；如果新系统开放使用后产生了业务数据，再决定回滚时必须先导出这些新增数据并人工确认处理方式，不能直接丢弃。

## 安全与运维

- PostgreSQL 只监听本机，移除旧系统后关闭对外开放的 `3306`。
- 数据库密码、JWT 密钥和迁移凭据均使用新随机值，只保存在权限为 `600` 的环境文件中。
- 源码包和发布包在使用前验证官方校验值或 Git 提交。
- 新后端不以 root 用户运行；应用目录、日志目录和数据库目录分别使用最小必要权限。
- PostgreSQL 每日执行压缩备份，保留 7 天，并定期完成一次真实恢复演练。
- Nginx、systemd、数据库和迁移日志不得记录认证秘密或用户密码哈希。

## 自动验证与业务验收

迁移工具采用测试先行方式覆盖成员拆分、时区转换、权限映射、日志转换、序列重置、空表、软删除、源数据缺失、目标库非空和事务回滚。测试使用固定夹具，不连接正式数据库。

发布前运行项目根目录 `node scripts/verify-change.mjs`、数据库迁移检查、后端测试、前端构建和浏览器冒烟验收。生产环境安装后验证 Node.js、PostgreSQL、systemd、Nginx、磁盘空间、数据库备份、健康检查和实际入口。

最终业务验收覆盖管理员和业务权限用户登录，以及产品、项目、需求、任务的列表、查询、详情、历史记录和新增操作。用户确认验收后，正式状态才从“待发布确认”变为“已发布”。

## 不在本次范围

- 重新实现里程碑业务功能。
- 迁移或重新实现 MCP/Agent。
- 改造首页统计或新增业务模块。
- 修改既有产品、项目、需求、任务的业务规则。
- 清理旧服务器上的 MySQL、旧应用或旧备份；清理在新系统稳定运行 30 天后单独确认。
