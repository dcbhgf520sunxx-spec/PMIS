# 项目合同及阶段付款设计

## 目标

在现有项目管理中增加供应商合同和阶段付款能力。一份项目最多维护一份供应商合同；合同包含多个手工定义的付款阶段；每个付款阶段允许登记多笔实际付款。

本期只管理付款计划和实际付款登记，不增加付款申请、审批、发票、供应商档案、附件、预算或财务报表。

## 已确认业务口径

- 管理我方向供应商支付的项目支出。
- 一个项目最多一份合同。
- 付款阶段由用户在合同中手工维护。
- 一个付款阶段允许分多次付款。
- 不增加独立合同菜单，功能继续放在“项目管理”中。
- 不增加合同或付款动作权限；拥有现有“项目管理”菜单权限的用户即可使用全部合同和付款功能。
- 项目详情复用现有 `TemplateDetailPage.sectionNavigation` 分类导航能力，顶部只设置“基本信息”和“合同信息”两个分类入口，不扩展公共详情组件。

## 页面设计

### 项目详情

项目详情保留现有标题、项目状态、逾期状态、状态操作和单据信息侧栏。主内容启用 `sectionNavigation`：

- “基本信息”对应 `sectionKey="basic"`，包含现有项目基本信息、进展与风险及项目变更历史。
- “合同信息”对应 `sectionKey="contract"`，包含合同信息和付款阶段。

这里复用的是现有详情分类导航：点击分类后定位到对应详情区域，不新增一套切换并隐藏内容的业务 Tab。

项目没有合同时，“合同信息”区域使用统一空状态并提供“新增合同”。已有合同时，合同区域提供“编辑合同”。新增和编辑合同进入独立表单页，返回时回到原项目详情上下文。

### 合同信息

合同详情展示以下字段：

| 字段 | 来源 | 规则 |
|---|---|---|
| 所属项目 | 系统带入 | 一份合同固定关联当前项目，不允许手工切换 |
| 合同编码 | 用户录入 | 必填，未删除合同中唯一 |
| 合同名称 | 用户录入 | 必填 |
| 供应商 | 用户录入 | 必填，本期使用文本，不建设供应商档案 |
| 签订时间 | 用户录入 | 必填，精确到日 |
| 合同金额 | 用户录入 | 必填，大于 0，保留两位小数 |
| 已付金额 | 系统计算 | 合同下全部有效付款记录之和，只读 |
| 未付金额 | 系统计算 | 合同金额减已付金额，只读 |

合同表单同时维护合同字段和付款阶段。合同及阶段通过同一接口、同一数据库事务保存，任何校验失败都不能产生部分数据。

### 付款阶段

付款阶段使用 `TemplateDetailTableSection` 展示，不在业务页直接拼表格。

| 字段 | 来源 | 规则 |
|---|---|---|
| 阶段 | 用户录入 | 必填，同一合同内名称不重复 |
| 计划金额 | 用户录入 | 必填，大于 0，保留两位小数 |
| 已付金额 | 系统计算 | 当前阶段有效付款记录之和，只读 |
| 待付金额 | 系统计算 | 计划金额减已付金额，只读 |
| 付款状态 | 系统计算 | 未付款、部分付款、已付清 |
| 操作 | 系统提供 | 登记付款、付款明细 |

付款状态计算规则：

- 已付金额等于 0：未付款。
- 已付金额大于 0 且小于计划金额：部分付款。
- 已付金额等于计划金额：已付清。

本期没有计划付款时间，因此不计算付款逾期。所有付款阶段的计划金额合计必须等于合同金额。

已付清的阶段不再显示“登记付款”，只显示“付款明细”。付款明细使用现有标准抽屉表格能力，展示该阶段的全部实际付款记录。

### 登记付款

在付款阶段操作列点击“登记付款”，使用现有后台表单和弹窗能力填写：

| 字段 | 来源 | 规则 |
|---|---|---|
| 本次付款金额 | 用户录入 | 必填，大于 0，不能超过当前阶段待付金额 |
| 付款时间 | 用户选择 | 必填，选择和展示到月，格式为 `YYYY-MM` |
| 经办人 | 用户选择 | 必填，从现有系统用户中选择 |
| 备注 | 用户录入 | 选填，多行文本 |

付款时间在数据库中使用 `DATE`，统一保存为所选月份的第一天，页面仅按月选择和展示。实际付款月份不能晚于当前月份。

付款明细同时展示登记人和登记时间。允许通过付款明细编辑或删除误登记；删除采用逻辑删除，并立即重新计算阶段和合同金额。

## 核心业务规则

1. 一个项目最多存在一份未删除合同。
2. 合同编码在未删除合同中唯一。
3. 合同金额和所有阶段计划金额均使用两位小数精确计算，不使用浮点数。
4. 付款阶段计划金额合计必须等于合同金额。
5. 单笔付款金额不得超过当前阶段待付金额。
6. 阶段累计已付金额不得超过阶段计划金额，合同累计已付金额不得超过合同金额。
7. 已有付款记录的阶段不能删除；修改计划金额后不得小于该阶段已付金额。
8. 修改合同金额时，阶段计划金额合计必须同步等于新合同金额，且合同金额不得小于累计已付金额。
9. 删除实际付款后，阶段已付金额、待付金额、付款状态以及合同已付金额、未付金额必须在同一请求结果中重新计算。
10. 合同已存在时不能重复新增；本期不提供合同删除，只允许编辑合同和阶段。
11. 项目存在合同时不能删除，接口返回明确的业务提示。
12. 合同新增、编辑、登记付款、修改付款和删除付款均写入现有操作日志，操作者从当前登录用户取得。

## 数据模型

### `pms_project_contract`

存储项目合同，一对一关联项目。

主要字段：

- `id BIGSERIAL PRIMARY KEY`
- `project_id BIGINT NOT NULL REFERENCES pms_project(id) ON DELETE RESTRICT`
- `contract_code VARCHAR(100) NOT NULL`
- `contract_name VARCHAR(200) NOT NULL`
- `supplier_name VARCHAR(200) NOT NULL`
- `signed_date DATE NOT NULL`
- `contract_amount NUMERIC(18,2) NOT NULL`
- `creator_id`、`updater_id`、`is_deleted`、`created_at`、`updated_at`

约束和索引：

- 对未删除数据建立 `project_id` 唯一索引，保证一项目一合同。
- 对未删除数据建立 `contract_code` 唯一索引。
- `contract_amount > 0`。

### `pms_project_payment_stage`

存储合同付款阶段。

主要字段：

- `id BIGSERIAL PRIMARY KEY`
- `contract_id BIGINT NOT NULL REFERENCES pms_project_contract(id) ON DELETE RESTRICT`
- `stage_name VARCHAR(100) NOT NULL`
- `planned_amount NUMERIC(18,2) NOT NULL`
- `sort_order INTEGER NOT NULL`
- `creator_id`、`updater_id`、`is_deleted`、`created_at`、`updated_at`

约束和索引：

- 未删除阶段在同一合同内名称唯一。
- `planned_amount > 0`。
- 按 `contract_id、sort_order、id` 稳定排序。

### `pms_project_payment_record`

存储实际付款记录，一个阶段允许多条。

主要字段：

- `id BIGSERIAL PRIMARY KEY`
- `stage_id BIGINT NOT NULL REFERENCES pms_project_payment_stage(id) ON DELETE RESTRICT`
- `payment_amount NUMERIC(18,2) NOT NULL`
- `payment_month DATE NOT NULL`
- `handler_id BIGINT NOT NULL REFERENCES pms_user(id) ON DELETE RESTRICT`
- `remark TEXT`
- `creator_id`、`updater_id`、`is_deleted`、`created_at`、`updated_at`

约束和索引：

- `payment_amount > 0`。
- `payment_month` 统一为月份第一天。
- 为 `stage_id、is_deleted、payment_month` 建立查询索引。

已付金额、未付金额、待付金额和付款状态不落冗余字段，由有效付款记录实时聚合，避免付款更正后产生汇总不一致。

## 接口设计

- `GET /projects/:projectId/contract`：查询合同、付款汇总和付款阶段；无合同时返回明确的空结果。
- `POST /projects/:projectId/contract`：新增合同及全部付款阶段。
- `PUT /projects/:projectId/contract`：编辑合同及付款阶段。
- `GET /projects/:projectId/contract/stages/:stageId/payments`：查询阶段付款明细。
- `POST /projects/:projectId/contract/stages/:stageId/payments`：登记付款。
- `PUT /projects/:projectId/contract/payments/:paymentId`：更正付款。
- `DELETE /projects/:projectId/contract/payments/:paymentId`：逻辑删除误登记付款。

接口统一使用现有 `{ code, message, data }` 响应结构。合同与阶段表单校验错误使用结构化 `fieldErrors`；金额上限、关联关系等业务规则必须由后端重新校验，不能只依赖前端。

所有接口继续使用现有项目菜单权限 `project`，不增加新的权限键或菜单。

## 变更历史

项目原有变更历史继续展示项目字段变化。合同和付款操作也写入现有 `pms_op_log`：

- 合同新增和编辑以“项目合同”为模块记录。
- 付款新增、修改和删除以“项目付款”为模块记录。
- 同次合同保存涉及多个合同或阶段字段时，共享同一个 `operation_id`。
- 展示时将合同、阶段、经办人等关联值转译为中文名称，不展示数据库原始 ID。

合同和付款历史本期合并展示在项目详情现有“变更历史”中，不新增独立历史页面。

## 实施边界

实施时遵循现有业务页面样板和交付链路，主要涉及：

- 项目详情页、合同表单页、付款登记及付款明细。
- 项目前端 API、运行时响应契约和类型。
- 项目合同、付款阶段和实际付款后端接口。
- 初始化 SQL、增量 migration、后端查询和测试。
- `docs/数据库表结构.md` 和 `docs/数据库表结构.xlsx`。
- 路由、菜单访问校验、组件审计和交付门禁。

不新增独立合同菜单，不新增权限，不修改公共详情模板，不建设供应商档案、审批、发票、附件、预算、提醒或报表。

## 验收标准

- 项目详情顶部使用现有分类导航显示“基本信息”和“合同信息”。
- 无合同时展示统一空状态；同一项目不能重复创建合同。
- 合同字段和付款阶段全部读取真实接口及数据库数据。
- 阶段计划金额合计不等于合同金额时无法保存。
- 同一阶段能够登记多笔付款，付款时间按月选择和展示。
- 超过阶段待付金额的付款被后端拦截。
- 阶段已付、待付、付款状态以及合同已付、未付金额计算正确。
- 修改或删除付款后全部汇总立即正确回算。
- 只有现有项目菜单权限校验，不出现新增权限键或独立菜单。
- 数据库初始化 SQL、migration、结构 Markdown 和结构 Excel 保持一致。
- 相关单元测试、接口测试、前端构建、项目完整门禁和实际浏览器流程验收通过。

## 实施前确认点

本设计包含三张新增数据库表。按照项目交付规则，进入实现前需要用户确认本设计中的表结构，再由 AI 编写并执行 migration。
