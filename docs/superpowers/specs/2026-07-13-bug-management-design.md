# BUG 管理迁移设计

## 范围

将 `project-manage-system` 的 BUG 管理完整迁入 PMIS。不迁移旧系统历史数据，不修改首页，新增 60 条可验收的模拟 BUG 及对应变更历史。业务规则沿用源系统，页面、接口、权限、日志和数据库实现遵守 PMIS 当前 React、Express、PostgreSQL、页面样板和公共组件规范。

## 数据模型

新增 `pms_bug`，字段包含：主键、关联类型、关联项目、关联需求、BUG 标题、富文本 BUG 描述、BUG 类型档案 ID、严重程度、状态、指派人、解决方案档案 ID、修复时间、关闭时间、创建人、更新人、软删除和创建/更新时间。

- `source_type` 只允许 1 项目、2 需求，且 `project_id` 与 `requirement_id` 必须二选一。
- `bug_type_id` 关联基础档案类型“Bug类型”；`resolution_id` 关联基础档案类型“Bug解决方案”，只在已修复或已关闭时可保留。
- 严重程度为 1 低、2 中、3 高、4 致命；状态为 0 新建、1 已修复、2 已关闭、3 被激活。
- 未删除 BUG 标题唯一；复制页原样回填标题，未修改保存时由现有唯一校验报错。
- 常用的关联、指派、状态、严重程度、类型和创建时间字段建立索引。历史复用 `pms_op_log.operation_id` 聚合，不新增历史表。

## 基础档案

新增两个档案类型：

- `bug_type` / `BT` / `Bug类型`：功能、界面、性能、兼容、安全、其他。
- `bug_resolution` / `BR` / `Bug解决方案`：已修复、设计如此、无法重现、延期处理、外部原因、重复。

表单、筛选、详情、历史和模拟数据均通过档案 ID 与档案名称使用这两组值，不在业务页写死选项。已被有效 BUG 引用的档案不允许删除。

## 状态与时间规则

- 新建 → 已修复、已关闭。
- 已修复 → 已关闭、被激活。
- 已关闭 → 被激活。
- 被激活 → 已修复。
- 转为已修复必填修复时间和解决方案。转为已关闭必填关闭时间；已有修复信息可保留。
- 离开已修复/已关闭时清理不再适用的修复、关闭和解决方案字段。
- 页面、提示、历史和文档统一使用“时间”命名，数据库字段使用 `*_date` 不改变用户可见语义。

## 页面与交互

- 列表使用 `TemplateListPage`，提供全部 BUG / 我的 BUG 真实计数、组合筛选、服务端排序分页、加载/错误/重试状态。业务列为 BUG 标题、关联对象、指派给、BUG 类型、严重程度、状态、创建人、创建时间。
- 关联类型复用任务管理的项目/需求分类标签及底座色彩规范。严重程度和状态使用标签。
- 单条操作按编辑、状态变更、复制、删除排列。批量操作包含批量指派、同状态批量变更和批量删除。
- 新增/编辑/复制使用 `TemplateFormPage`；BUG 标题和富文本 BUG 描述依次独占一行；关联项目/需求二选一；指派给位于“处理信息”第一项。
- 详情使用 `TemplateDetailPage`，标题和右侧当前状态同步显示 BUG 状态和严重程度；状态操作紧随当前状态；提供基本信息、处理信息、变更历史、单据信息和保持列表筛选排序的上一条/下一条。

## 接口、权限与模拟数据

接口包含列表、详情、新增、编辑、删除、状态变更、批量指派、历史、上下条、标题唯一校验和权限内项目/需求选项。所有写操作只使用 `req.user.id` 作为操作人，并记录可读的中文变更历史。

新增 `/bugs` 独立菜单和 `bug` 权限，排在任务管理之后，新增/复制/编辑/详情子路由继承该权限。

模拟数据 migration 在现有项目、需求、启用用户和基础档案中选择有效关联，生成 60 条 BUG，覆盖全部类型、严重程度、状态和解决方案。每条至少生成“新增”日志，非新建状态增加相应状态变更日志。migration 可重复执行且不会重复插入。

## 交付与验收

同步初始化 SQL、增量 migration、后端 SQL/接口/测试、数据库 Markdown 与 Excel、前端页面和权限。执行完整自动门禁、数据库迁移、真实接口读写以及 3104 浏览器列表/表单/详情/状态/批量/导航验收；最后确认 3103 健康检查和 3104 页面可访问。
